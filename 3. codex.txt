# AGENTS.md

## Code Style
- Use Black for Python formatting.
- Avoid abbreviations in variable names.

## Testing
- Run pytest tests/ before finalizing a PR.
- All commits must pass lint checks via flake8.

## PR Instructions
- Title format: [Fix] Short description
- Include a one-line summary and a "Testing Done" section

- We can write configurations in .codex/config.toml if want to share same cofig across team with default options.
- We can set rules in ./codex/rules/ to control which commands codex can run outside the sandbox.

- MCP: Model Context protocol
- It connects models to tools and context.
- Configure in config.toml
- Like connect with figma, notion
- Useful MCP servers: 
Context7: Connect to up-to-date developer documentation.
Figma Local and Remote: Access your Figma designs.
Playwright: Control and inspect a browser using Playwright.
Chrome Developer Tools: Control and inspect Chrome.
Sentry: Access Sentry logs.
GitHub: Manage GitHub beyond what git supports (for example, pull requests and issues).

- Skills:
- Give Codex new capabilities and expertise.
- A skill packages instructions, resources, and optional scripts so Codex can follow a workflow reliably.
- You can share skills across teams or with the community. 
- Can create: .codex/skills: SKILL.md, scripts/, references/, assets/
- To select one, run the /skills slash command, or start typing $ to mention a skill. 
- Can enable or disable skill in config file.

Skill instructions for the Codex agent to follow when using this skill.
- We can install or download skills from other repositories also.
- Use skills when you want to share behavior across a team, enforce consistent workflows, or encode best practices once and reuse them everywhere.
- Create a Skill: $skill-creator, promt: Create a skill that drafts a conventional commit message based on a short summary of changes.
- or manually in .codex/skills/skill-name:
---
name: draft-commit-message
description: Draft a conventional commit message when the user asks for help writing a commit message.
metadata:
  short-description: Draft an informative commit message.
---

Draft a conventional commit message that matches the change summary provided by the user.

Requirements:

- Use the Conventional Commits format: `type(scope): summary`
- Use the imperative mood in the summary (for example, "Add", "Fix", "Refactor")
- Keep the summary under 72 characters
- If there are breaking changes, include a `BREAKING CHANGE:` footer
